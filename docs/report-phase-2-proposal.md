# Phase 2 리포트 화면 제안

작성일: 2026-06-25
상태: 검토 완료 — 1차 범위 확정 (Claude Code 의견 반영)
관련 구현 PR: `feat/report-phase-2-v1`

## 목적

Phase 2의 리포트 화면은 저장 개수를 보여주는 통계 화면이 아니라, 사용자가 "요즘 내가 무엇에 관심을 두고 있는지"를 확인하는 화면으로 정의한다.

Nook의 핵심 가치는 콘텐츠 저장 자체보다 저장된 흔적을 다시 읽어주는 데 있다. 따라서 리포트는 단순 저장량보다 관심 카테고리, 관심 흐름, 관심 분포, AI 피드백을 우선한다.

## 1차 제안 화면 구조

```text
리포트

┌─ AI 코멘트 ───────────────────┐
│ 최근에는 AI 자동화와 디자인    │
│ 관련 콘텐츠가 자주 저장됐어요. │
│ 특히 Agent, Workflow, UX 같은 │
│ 주제가 반복해서 보여요.        │
└─────────────────────────────┘

┌─ 많이 모은 카테고리 ───────────┐
│ AI              12개           │
│ 디자인           8개           │
│ 커리어           5개           │
└─────────────────────────────┘

┌─ 관심사 흐름 ─────────────────┐
│ 최근 늘어난 관심              │
│ AI Agent / Workflow / UX       │
│                               │
│ 최근 줄어든 관심              │
│ 경제 / 여행                    │
└─────────────────────────────┘

┌─ 관심 분포 ───────────────────┐
│ AI              42%            │
│ 디자인          31%            │
│ 커리어          18%            │
│ 기타             9%            │
└─────────────────────────────┘

┌─ 관련 주제 ───────────────────┐
│ AI: Agent, Automation, Workflow│
│ 디자인: UX, Product, Interior │
│ 커리어: Portfolio, Interview  │
└─────────────────────────────┘
```

## 핵심 판단

### 저장 개수는 보조 정보로 둔다

사용자는 "몇 개 저장했는지"보다 "요즘 어떤 주제에 끌리고 있는지"를 더 궁금해할 가능성이 높다. 저장 개수는 카테고리와 주제의 근거로만 표시한다.

예:
- `AI 12개`
- `Agent 5회 등장`

### 태그 단독 섹션은 약하게 둔다

태그는 카테고리와 겹칠 수 있다. 특히 현재 AI 분류 구조가 `category + tags`를 함께 반환하므로 다음처럼 중복될 수 있다.

```text
category: AI
tags: AI, Agent, Workflow
```

따라서 태그를 독립된 큰 섹션으로 두기보다 카테고리 안의 세부 주제 근거로 사용한다.

추천:
- `자주 등장한 태그` 단독 섹션보다 `관련 주제`
- 카테고리별 하위 키워드로 표현
- AI 코멘트의 근거 데이터로 활용

## 리포트 주기

기본 정책은 주간 리포트 + 월간 리포트 보조를 제안한다.

### 주간 리포트

- 기본 화면 기준: 최근 7일
- 목적: 요즘 관심 변화 확인
- AI 코멘트 우선 적용 대상
- 저장 패턴이 빠르게 변하는 사용자에게 적합

생성 조건:
- 최근 7일 저장 콘텐츠 3개 이상

부족 상태:

```text
아직 이번 주 관심사를 읽기엔 기록이 조금 적어요.
조금 더 저장하면 흐름을 보여드릴게요.
```

### 월간 리포트

- 기준: 최근 30일
- 목적: 큰 관심 분포와 장기 흐름 확인
- 초기에는 단순 집계 중심
- 이후 공유/회고/월말 리캡으로 확장 가능

생성 조건:
- 최근 30일 저장 콘텐츠 8개 이상

### 재생성 정책

AI 코멘트는 매 화면 진입마다 생성하지 않는다.

추천:
- 조건 충족 시 1회 생성 후 캐싱
- 같은 주기 안에서 새 콘텐츠 3개 이상 추가되면 재생성 가능
- 다음 주기 진입 시 새 리포트 생성
- 중복 URL 저장은 기존 정책에 따라 새 리포트 트리거에 포함하지 않음

## AI 코멘트 제어 원칙

AI 코멘트는 자유로운 해석보다 관찰 기반 피드백이어야 한다.

### 출력 형식 제안

```json
{
  "headline": "최근에는 AI 자동화와 디자인 주제가 자주 보여요.",
  "observations": [
    "AI 카테고리에서는 Agent, Workflow 태그가 반복됐어요.",
    "디자인 카테고리 저장이 지난 기간보다 늘었어요."
  ],
  "suggestion": "이번 주에는 저장해둔 AI Agent 글을 다시 열어보면 좋아요."
}
```

### 허용되는 표현

- `자주 저장됐어요`
- `반복해서 보여요`
- `지난 기간보다 늘었어요`
- `최근에는 ... 쪽으로 관심이 모이는 것 같아요`

### 피해야 할 표현

- `당신은 ...한 사람입니다`
- `당신의 성향은 ...입니다`
- 저장 데이터에 없는 관심사 추론
- 건강, 재정, 정치 성향 등 민감한 추론
- 개별 콘텐츠 요약
- 앱 밖 행동을 유도하는 조언

### 제안 문구 범위

AI 제안은 앱 안 행동으로 제한한다.

가능:
- 저장해둔 콘텐츠 다시 열어보기
- 특정 카테고리 훑어보기
- 오래 열지 않은 콘텐츠 확인하기
- 카테고리 정리하기

불가능:
- 새로운 검색 키워드 추천
- 외부 학습 계획 제안
- 사용자의 직업/성향/생활 패턴 추정

## 데이터 근거

리포트는 아래 데이터만 사용한다.

- `contents.category_id`
- `contents.tags`
- `contents.saved_at`
- `contents.viewed_at`
- `contents.domain`
- 현재 사용자의 `categories`

보안 원칙:
- 모든 쿼리는 `user_id`로 스코프한다.
- 다른 사용자의 데이터는 리포트 생성 입력에 포함하지 않는다.
- AI 입력에는 필요한 최소 데이터만 전달한다.
- 가능하면 전체 URL보다 `title`, `domain`, `category`, `tags`, `saved_at` 중심으로 전달한다.

## 구현 방향 초안

### 1단계: 집계 기반 리포트

- 많이 모은 카테고리
- 관심 분포
- 관련 주제
- 저장량 부족 상태

이 단계는 AI 호출 없이 구현 가능하다.

### 2단계: AI 코멘트 추가

- 집계 결과를 입력으로 AI 코멘트 생성
- 저장 데이터 원문 전체를 넘기기보다 요약된 집계 데이터 전달
- 프롬프트 파일은 별도 버전 관리

추천 경로:

```text
prompts/report/
├── v1.txt
└── eval.ts
```

### 3단계: 캐싱과 주기 관리

검토 필요:
- `reports` 테이블을 새로 둘지
- Supabase Edge Function에서 생성할지
- 앱 클라이언트에서 집계 후 AI 호출을 요청할지

가능한 캐시 스키마 초안:

```text
reports
- id
- user_id
- period_type: weekly | monthly
- period_start
- period_end
- input_hash
- result_json
- generated_at
- created_at
- updated_at
```

주의:
- RLS 필수
- `unique(user_id, period_type, period_start, period_end)` 검토
- 같은 입력에 대한 중복 AI 호출 방지

## Claude Code 검토 요청 사항

1. 리포트 데이터를 클라이언트에서 직접 집계할지, Supabase RPC/Edge Function으로 분리할지
2. AI 코멘트를 저장할 `reports` 테이블이 필요한지
3. 주간/월간 기간 계산을 KST 기준으로 할지, 사용자 locale 기준으로 할지
4. AI 입력 데이터 최소화 기준이 적절한지
5. 태그와 카테고리 중복을 줄이기 위한 집계/프롬프트 규칙
6. `prompts/report/v1.txt`를 classification prompt와 분리하는 방향의 적절성
7. Phase 2에서 먼저 구현할 최소 범위

## 현재 추천 최소 범위

첫 구현은 아래 범위로 제한한다.

1. 리포트 탭 UI 개편
2. 최근 7일 기준 집계
3. 많이 모은 카테고리
4. 관심사 흐름
5. 관심 분포
6. 관련 주제
7. 저장량 부족 상태

AI 코멘트는 프롬프트와 캐싱 전략을 Claude Code가 검토한 뒤 별도 작업으로 진행한다.

## 최종 결정 (Claude Code 검토 반영, 2026-06-25)

### Phase 2 리포트 1차 범위

1. **많이 모은 카테고리 Top 3+** — `category_id IS NOT NULL`만 집계, 미분류 제외
2. **관심 분포** — 카테고리별 비율(%) 바
3. **관련 주제** — 카테고리당 자주 등장한 태그. **category 이름과 같은 tag는 제외** (예: `category=AI, tag=AI`는 표시 X)
4. **부족 상태 안내** — fallback도 부족하면 "조금 더 저장하면..." 안내
5. **미분류 안내** — Top 카테고리에는 포함하지 않고 별도 줄로 "분류되지 않은 콘텐츠 N개"

### 기간 정책 — 7일 우선, 30일 fallback

| 조건 | 동작 | UI 표시 |
|---|---|---|
| 7일 분류된 콘텐츠 ≥ 3 | 7일 기준 표시 | 상단 "최근 7일 기준" |
| 7일 < 3 AND 30일 분류된 ≥ 3 | 30일 fallback | 상단 "최근 한 달 기준" + 첫 진입 시 작은 배너 |
| 30일 < 3 | 부족 상태 안내 | "기록이 더 모이면 보여드릴게요" |

### 1차 제외 (2차 별도 PR)

- **관심사 흐름** (week-over-week diff) — 출시 직후 비교 기준 데이터 부족, "늘었다/줄었다"가 오히려 신뢰 깎을 수 있음
- **AI 코멘트** (`prompts/report/v1.txt`) — 비용/캐싱/재생성 정책 검토 후 별도 PR
- **reports 테이블 캐싱** — AI 코멘트 도입 시 함께
- **주간/월간 전환** — 사용 데이터 보고 필요성 판단

### 기술 결정

- **AI 호출 없음** — 1차는 집계만
- **reports 테이블 없음** — 단일 사용자 콘텐츠 수십~수백 개라 매 진입 시 클라이언트 집계로 ms 단위
- **단일 fetch + 클라이언트 derive** — `getRecentContentsForReport(days)` 한 번 호출 후 카테고리/분포/주제 모두 derive (중복 쿼리 회피)
- **KST 고정** — analytics-queries 패턴과 일관, timezone 분기 없음
- **태그-카테고리 중복 제거** — category 이름과 같은 tag는 관련 주제에서 제외

### 보안 / 데이터 원칙

- 모든 쿼리 `user_id` 스코프
- 단일 fetch는 `categories(name)` join + `category_id IS NOT NULL` 필터
- 1차에는 AI 입력 없음 (외부 전송 데이터 0)

### 구현 순서

1. `lib/api.ts`에 `getRecentContentsForReport(days)` 추가
2. `lib/report.ts` 신규 — 집계 함수 (`aggregateByCategory`, `computeDistribution`, `topTagsPerCategory`)
3. `app/(tabs)/report.tsx` 전면 개편 — 4개 카드 + fallback 로직 + 기간 표시

### Phase 2 고도화 (별도 트랙)

- 관심사 흐름 (week-over-week)
- AI 코멘트 + `prompts/report/v1.txt`
- `reports` 테이블 + 캐싱
- 주간/월간 전환
